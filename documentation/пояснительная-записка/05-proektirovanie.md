# Проектирование системы

Проектирование системы – это процесс определения структуры, функций, данных, связей и правил работы будущей системы до начала программной реализации [10].

На этапе проектирования системы необходимо определить основные принципы построения программного решения и подготовить его к дальнейшей реализации. В процессе проектирования будет выполнен анализ требований, определена структура системы, а также разработаны основные компоненты и механизмы взаимодействия пользователей с системой.

На данном этапе жизненного цикла будут решены следующие задачи:

- анализ и уточнение требований к разрабатываемой системе;

- определение общей архитектуры системы;

- проектирование структуры базы данных;

- определение ролей пользователей и разграничение прав доступа;

- разработка логики работы системы удалённого оформления заказов;

- проектирование пользовательского интерфейса системы;

- проектирование механизма обработки и мониторинга заказов;

- разработка алгоритмов взаимодействия менеджеров и курьеров при доставке.

#### Выбор и обоснование архитектуры системы

Архитектура программной системы представляет собой совокупность основных решений, определяющих структуру системы, её компоненты, способы их взаимодействия и принципы организации работы. Архитектура описывает, из каких частей состоит система, как эти части взаимодействуют между собой и каким образом обеспечивается выполнение всех требуемых функций [11].

- Существует несколько основных видов архитектуры программных систем.

- Клиент-серверная архитектура – модель построения информационной системы, при которой приложение разделено на две части: клиентскую и серверную. Клиент отвечает за взаимодействие с пользователем и отправляет запросы, а сервер принимает эти запросы, обрабатывает их, работает с базой данных и возвращает результат. Данная архитектура широко используется при разработке веб-сервисов и информационных систем.

- Монолитная архитектура – способ построения программного обеспечения, при котором все компоненты системы объединены в одно приложение. Интерфейс пользователя, бизнес-логика и работа с данными находятся внутри единой программы и работают как единое целое. Такая архитектура отличается простотой разработки и развертывания.

- Микросервисная архитектура – подход, при котором система состоит из набора независимых сервисов. Каждый сервис выполняет отдельную функцию и может разрабатываться, обновляться и масштабироваться независимо от других. Такой подход используется в крупных распределённых системах.

- Для систем, работающих через сеть, часто применяется клиент-серверная архитектура. В рамках данной модели существует несколько вариантов организации взаимодействия.

- Двухзвенная архитектура включает два основных уровня:

- клиентское приложение;

- сервер, который одновременно выполняет обработку данных и работу с базой данных.

- В такой архитектуре (представленной на рисунке 14) клиент отправляет запросы напрямую на сервер, а сервер выполняет всю логику обработки и возвращает результат.

- В данной модели клиент взаимодействует с сервером приложений, который обрабатывает запросы и обращается к базе данных. Такой подход повышает масштабируемость системы и облегчает её сопровождение.

- Разрабатываемая система удалённого создания заказов будет реализована в виде веб-приложений. Веб-приложение – это программная система, доступ к которой осуществляется через веб-браузер по сети интернет или локальной сети. Пользователь взаимодействует с системой через интерфейс веб-страницы, а обработка данных выполняется на сервере [12].

- Обмен данными между клиентом и сервером осуществляется с использованием сетевых протоколов. Протокол — это набор правил, определяющих формат и порядок передачи данных между устройствами в сети [13].

- Наиболее распространённым протоколом для веб-приложений является HTTP (HyperText Transfer Protocol). Он используется для передачи данных между клиентом и сервером. Клиент отправляет HTTP-запрос, содержащий необходимую информацию, а сервер обрабатывает его и возвращает HTTP-ответ. Для повышения безопасности часто используется протокол HTTPS, который обеспечивает шифрование передаваемых данных.

- Также в информационных системах выделяют различные типы клиентов [14].

- “Толстый” клиент – это приложение, в котором значительная часть обработки данных выполняется на стороне пользователя. Такие программы устанавливаются на компьютер пользователя и могут работать с сервером только для обмена данными.

- “Тонкий” клиент – это приложение, в котором основная обработка данных выполняется на сервере. Пользователь взаимодействует с системой через веб-браузер или простое клиентское приложение. Такой подход снижает требования к устройствам пользователей и упрощает обновление системы.

- Разрабатываемая система удалённого создания заказов будет реализована как веб-приложение с использованием клиент-серверной архитектуры. В системе будет применяться двухзвенная архитектура. Взаимодействие между клиентом и сервером будет осуществляться по протоколу HTTP/HTTPS. В качестве клиента будет использоваться “тонкий” клиент – веб-браузер пользователя.

#### Структурная схема системы

Сущность структурного подхода к разработке информационных систем заключается в представлении сложной системы в виде иерархической структуры взаимосвязанных компонентов. При использовании данного подхода система последовательно декомпозируется, то есть разбивается на более простые части. Сначала выделяются основные функциональные подсистемы, затем они разделяются на подфункции, далее — на отдельные задачи и процедуры. Такое поэтапное разбиение продолжается до тех пор, пока каждая часть системы не станет достаточно простой для анализа и реализации.

Декомпозиция позволяет упростить процесс проектирования, так как разработка системы осуществляется по принципу «от общего к частному». При этом сохраняется целостность системы, поскольку все её элементы взаимосвязаны и направлены на выполнение общей задачи. Структурный подход также способствует повышению наглядности проектируемой системы и облегчает её дальнейшее развитие и сопровождение.

Система в общем случае представляет собой совокупность взаимосвязанных элементов, находящихся в отношениях и образующих единое целое. [?] Эти элементы взаимодействуют между собой для достижения определённой цели и выполнения заданных функций. В контексте информационных систем такими элементами являются программные компоненты, данные, технические средства и пользователи системы.

На рисунке 15 приведена структурная схема разрабатываемой системы, в ее состав входят следующие подсистемы:

1. подсистема пользовательского интерфейса, которая отвечает за отображение информации пользователю, включая визуализацию меню, отображение корзины, карточек блюд и текущего статуса заказа;

1. подсистема взаимодействия с серверной частью, которая обеспечивает обмен данными между клиентской и серверной частями системы посредством HTTP-запросов и передачу данных в формате JSON;

1. подсистема ввода данных, которая отвечает за ввод и обработку пользовательских данных, включая регистрацию и авторизацию пользователей, оформление заказов и управление личным кабинетом;

1. справочная подсистема, которая предоставляет пользователям информацию о работе системы и справочные материалы;

1. подсистема контроллеров, которая принимает HTTP-запросы от клиентской части системы, обрабатывает их и передаёт соответствующим компонентам серверной части;

1. подсистема взаимодействия с клиентской частью, которая обеспечивает обработку запросов от клиентов и формирование ответов, передаваемых обратно в клиентское приложение;

1. подсистема бизнес-логики, которая реализует основные функциональные возможности системы и обеспечивает обработку данных и выполнение операций над ними;

1. подсистема авторизации пользователей, которая выполняет проверку учетных данных пользователей и обеспечивает управление доступом к функциям системы;

1. подсистема управления меню, которая отвечает за формирование и редактирование меню блюд, а также за хранение и обновление информации о блюдах;

1. подсистема обработки заказов, которая обеспечивает создание, изменение и контроль состояния заказов пользователей;

1. подсистема управления пользователями, которая отвечает за хранение и обработку информации о пользователях системы;

1. подсистема управления доставкой, которая обеспечивает назначение заказов курьерам и контроль процесса доставки;

1. подсистема работы с базой данных, которая обеспечивает хранение и обработку данных системы, включая информацию о пользователях, блюдах, заказах и доставках.

#### Разработка прототипа интерфейса пользователя системы

Пользовательский интерфейс (user interface) – это совокупность программных и аппаратных средств, обеспечивающих взаимодействие пользователя с системой. Интерфейс определяет способы отображения информации, ввода данных и управления функциями приложения. Качество интерфейса напрямую влияет на удобство работы с системой и скорость освоения её возможностей пользователями [16].

При проектировании пользовательского интерфейса будут учитываться принципы удобства использования: последовательность и предсказуемость элементов управления, минимизация действий для достижения цели, визуальная иерархия информации и своевременная обратная связь. Разработка прототипов страниц позволит заблаговременно выявить проблемы в организации интерфейса до начала программной реализации, согласовать функциональность с требованиями заинтересованных сторон и снизить затраты на последующие доработки.

Система будет состоять из двух типов клиентских веб-приложений: десктопное веб-приложение для сотрудников ресторана (администратор, менеджер, курьер), работающее через браузер на настольном компьютере, и мобильное веб-приложение для клиентов, оптимизированное под экраны смартфонов. В данном разделе будут разработаны прототипы страниц обоих приложений.

Доступ к приложению будет осуществляться через страницу авторизации с вводом адреса электронной почты и пароля. Приложение будет адаптировано под различные роли: администратор, менеджер и курьер.

В правом верхнем углу будет расположена ссылка на страницу регистрации нового клиента.

Боковое навигационное меню будет включать логотип приложения Food Rush, две группы навигационных ссылок, а также служебные пункты. Модальные окна в прототипе изображены рядом с таблицей для наглядности; в реализованном приложении они будут открываться поверх содержимого страницы.

#### Разработка информационно-логического проекта системы

Навигационная модель системы описывает совокупность страниц приложения и допустимых переходов между ними в зависимости от роли пользователя. Она позволяет наглядно представить пользовательский путь (user flow) — последовательность действий, которую совершает пользователь при решении конкретной задачи в системе.

Разрабатываемая система будет поддерживать четыре роли: администратор, менеджер, курьер и клиент. Все роли будут проходить через единую точку входа — страницу авторизации. После успешной аутентификации система определит роль пользователя и перенаправит его в соответствующий раздел. Регистрация в системе будет доступна только для клиентов.

Сотрудники ресторана (администратор и менеджер) будут работать с десктопной версией веб-приложения, построенной по принципу единого рабочего пространства с боковым навигационным меню: разделы будут доступны независимо друг от друга, переход между ними не потребует соблюдения какой-либо последовательности. Курьер и клиент будут пользоваться мобильной версией приложения с нижней навигационной панелью.

Для каждой роли разработана отдельная навигационная модель. Сплошными стрелками обозначены прямые переходы вперёд по пользовательскому пути.

##### Язык UML

Для специфицирования (построения точных, недвусмысленных и полных моделей) системы и ее документирования используется унифицированный язык моделирования UML. Он позволяет формализовать структуру и поведение системы с помощью набора графических нотаций, понятных как разработчикам, так и другим участникам проекта. Использование UML способствует улучшению коммуникации в команде, упрощает анализ требований и обеспечивает наглядное представление архитектуры и логики работы системы.

##### Сценарии

Сценарий (scenario) – определенная последовательность действий, которая описывает действия актеров и поведение моделируемой системы в форме обычного текста [27].

В контексте языка UML сценарий используется для дополнительной иллюстрации взаимодействия актеров и вариантов использования.

Рассмотрим несколько сценариев.

Вариант использования «Установить статус работы»

Краткое описание. Позволяет курьеру изменить свой рабочий статус («Работает» / «Не работает») на текущий день через мобильное веб-приложение. Система обновляет доступность курьера для назначения новых заказов и уведомляет менеджера об изменении статуса.

Актант. Курьер.

Предусловия. Смартфон курьера включён, соединение с интернетом установлено. Курьер успешно авторизован в системе. Открыта страница «Мои заказы» или «Профиль» мобильного приложения. В интерфейсе отображается переключатель рабочего статуса (см. рисунок 33).

Основной поток событий.

### Курьер нажимает на переключатель статуса работы («Статус: работаю» / «Статус: выходной») на странице «Мои заказы» или в разделе «Профиль».

### Система отображает модальное окно подтверждения с текущим статусом и кнопками «Подтвердить», «Отмена».

Курьер нажимает на кнопку «Подтвердить».

А1: Курьер нажимает на кнопку «Отмена» или закрывает модальное окно.

Система проверяет наличие активных заказов, назначенных данному курьеру.
